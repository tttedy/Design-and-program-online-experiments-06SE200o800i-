How to Deal With Git Merge Conflict

This page shows a typical scenario where your fork conflicts with the teacher's updates, and how to resolve it using VS Code.

Setup: Teacher's repository (upstream) → Your fork (main) → Local clone.

Step 1: Teacher updates the original repository

Teacher adds an emoji to the greeting in greet.js:

function greet(name) {
    console.log(`👋 Hello, ${name}!`);
}

Step 2: You modify the same file in your fork

You add a default parameter and change the wording:

function greet(name = 'World') {
    console.log(`Hey, ${name}!`);
}

You commit and push this change to your fork.

Step 3: You want to bring teacher's updates into your fork

You run (in terminal or VS Code):


    git remote add upstream https://github.com/J2K101000101/Design-and-program-online-experiments-06SE200o800i-.git
    git fetch upstream
    git checkout main
    git merge upstream/main

Because both changed the same function—and the signature line differs—Git reports a merge conflict that includes the signature.

Step 4: Conflict markers in VS Code

After the merge attempt, greet.js looks like this:

<<<<<<< HEAD
function greet(name = 'World') {
    console.log(`Hey, ${name}!`);
=======
function greet(name) {
    console.log(`👋 Hello, ${name}!`);
>>>>>>> upstream/main
}

The conflict markers show:

  • <<<<<<< HEAD – your version (with default parameter and "Hey")
  • ======= – divider between your version and the incoming version
  • >>>>>>> upstream/main – the teacher's version (without default, with emoji)

Step 5: Resolving the conflict in VS Code

Inline actions

Click inside the conflicted area. Above the markers you'll see options:

  • Accept Current Change – replaces the whole conflicted block with your version (default + "Hey")
  • Accept Incoming Change – replaces the whole block with the teacher's version (no default, emoji)
  • Accept Both Changes – places both versions one after the other (not useful here as they'd be two separate function definitions)
  • Compare Changes – opens a diff view

Suppose you want to keep the emoji from the teacher and your default parameter. Neither automatic option gives that exact result, so you'll need to manually edit the file. For example:

  • Click "Accept Current Change" to keep your version, then replace the console.log line with the teacher's emoji version.
  • Or click "Accept Incoming Change" to get the teacher's version, then add the default parameter back to the function signature.
  • Or simply edit directly inside the conflicted block.

After manual editing, the desired combined code is:

function greet(name = 'World') {
    console.log(`👋 Hello, ${name}!`);
}

If you also want to keep your "Hey" wording (e.g., `👋 Hey, ${name}!`), you can write that manually.

Resolve -> Save -> Commit-> Sync

  • Resolve the conflict and save the file (Ctrl+S / Cmd+S).
  • Type a commit message (e.g., fix) in the Source Control input box.
  • Click the Commit button to commit the merge.
  • Click the Sync Changes button to push the commit to your fork.
Worry Free: You never push directly to the teacher's repository. All operations only affect your local clone and your fork.

Summary

  • Add the teacher's repo as upstream (once).
  • Fetch new commits: git fetch upstream.
  • Merge upstream/main into your local main.
  • Resolve conflicts using VS Code's inline tools and manual editing if needed.
  • Commit and push/sync to your fork.
  • ❗❗❗ In the future, please follow README.md: 3. Create Your Personal Workspace!

This workflow keeps your fork in sync with the teacher's updates while preserving all your own work.
